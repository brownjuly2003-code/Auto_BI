## Findings

**F1 [P2] `auto_bi/api/app.py:218-232` + `auto_bi/agent/machine.py:169-175` + `auto_bi/api/static/app.js:264-274`** — Упавшая сборка невосстановима без правки spec. После approve машина переходит в APPROVED; если builder упал, `build_status="failed"`, но phase остаётся APPROVED. Повторный `POST /approve` проходит гард `build_status == "building"`, но `agent.approve()` требует phase==APPROVE → RuntimeError → **409 «nothing to approve in phase approved»**. Воспроизведено in-proc: `approve → build fail → state {phase: approved, build_status: failed} → retry approve → 409`. UI усугубляет: обработчик SSE `error` заново включает кнопку «Пересобрать» — пользователь кликает и получает необъяснимый 409. Единственный обход — сделать содержательную правку словами (noop не помогает: phase остаётся APPROVED). Транзиентные ошибки Superset/CH — нормальный режим живого стенда, retry без правки — естественное действие. Рекомендация: в `approve`-эндпоинте разрешить повтор при `phase==APPROVED and build_status=="failed"` (взять `managed.agent.spec` без вызова `agent.approve()`), либо сделать `AgentSession.approve()` идемпотентным из APPROVED; добавить тест «build failed → re-approve → 202».

**F2 [P3] `auto_bi/api/sessions.py:88-112` (SessionManager.start)** — Два сцепленных дефекта старта: (а) ManagedSession регистрируется в реестре ДО `agent.start()` (долгий LLM-вызов), и сам `start` выполняется без `managed.lock` — параллельный `POST /reply` на этот session_id (id виден через store или угадывается ретраем клиента) попадёт в незащищённую машину одновременно с grounding; (б) если `agent.start()` бросает (LLMError на первом grounding) — клиент получает голый 500, а зомби-сессия в фазе INTAKE остаётся в реестре навсегда (плюс строка в store). Контракт «ошибка LLM = 200 + error» (зеркало F6) реализован для reply, но не для start. Рекомендация: регистрировать сессию после успешного `start` (или держать lock на время старта) и маппить LLMError старта в осмысленный ответ (200+error или 503), убирая запись из реестра.

**F3 [P3] `auto_bi/api/sessions.py:70-120`** — Реестр сессий не ограничен: ни TTL, ни капа, ни DELETE-эндпоинта. Каждая ManagedSession держит spec/report/буфер событий; долгоживущий `auto_bi serve` накапливает их бесконечно. Для single-user некритично, но это процесс «на каждый день» (цель фазы). Рекомендация: дешёвый LRU/TTL (например, кап N сессий с вытеснением терминальных) — durable-история и так в Store.

**F4 [P3] `auto_bi/api/sessions.py:53-67` (stream_events) + `app.py:290-299`** — SSE-генератор синхронный и в ожидании событий крутится в `Condition.wait(1.0)` без yield: GeneratorExit при дисконнекте клиента доставляется только на yield, поэтому отвалившийся клиент не освобождает поток anyio-threadpool, пока не придёт терминальное событие. Если build-поток завис (повисший Superset без таймаута на каком-то шаге) — поток подвешен навсегда; несколько таких — и threadpool (по умолчанию 40) истощается. Рекомендация: периодический heartbeat-yield (SSE-комментарий `: ping\n\n`) из ветки таймаута — дисконнект обнаружится на ближайшем yield.

**F5 [P3] `auto_bi/api/app.py` (все мутирующие эндпоинты) + `auto_bi/cli.py:_serve`** — API без аутентификации и anti-CSRF: `PATCH /model/...` пишет в `model.yaml` на диске, `POST /approve` запускает сборку в BI; `--host` позволяет 0.0.0.0. Для localhost-продукта v1 приемлемо (дефолт 127.0.0.1), но cross-site запрос с сайта в том же браузере (form text/plain / DNS rebinding) может дотянуться до мутирующих ручек. Рекомендация (дёшево): проверять Host/Origin == localhost в middleware, задокументировать «serve — только localhost до Phase 4 (auth там и запланирован)».

**F6 [P3] `auto_bi/api/app.py:131-176` (update_column) — связка с F1** — Смена роли колонки measure→dimension через enrichment UI при живой сессии с уже предложенным spec (SUM по этой колонке) делает spec невалидным; «правки видны следующему grounding» — by design и задокументировано, но approve spec не перевалидирует, и ошибка всплывёт только на сборке (`compile_and_build` → defensive validate → SSE error) — а после F1 эта сборка ещё и невосстановима. Рекомендация: при approve перевалидировать spec против текущей модели и возвращать контрактную ошибку (200+error в reply-стиле или 409 с понятным detail) вместо падения в build-логе.

**F7 [P3] `auto_bi/semantic/dbt_import.py:52-54` (_node_table)** — Для dbt **sources** физическое имя таблицы — `identifier` (фоллбек `name`), а не `alias`; `_node_table` берёт `alias or name`, поэтому source с кастомным `identifier` не сматчится с моделью и уйдёт в `unmatched_models` (ложный репорт, обогащение потеряно; данные не портятся — политика «не добавлять» спасает). Рекомендация: `node.get("identifier") or node.get("alias") or node.get("name")` + тест на source с identifier.

**F8 [P3] `auto_bi/api/static/index.html:18-21` (#bi-target)** — «Селектор целевой BI» из задачи 2.2 декоративный: единственная опция Superset и значение нигде не читается (`app.js` его не отправляет, `target_bi` приходит из spec LLM/дефолта IR). С v1=Superset-only это честно по скоупу, но мёртвый контрол вводит в заблуждение. Рекомендация: либо прокидывать значение в сессию (заготовка под Phase 3), либо заменить статичным бейджем «Superset» до появления второй BI.

**Plan-deviations (зафиксированы, не баги)** — (1) Fields-first: вместо «вариантов дашборда» из §3.7 — один spec + детерминированный `seed_analysis`; отклонение явно задокументировано в ARCHITECTURE §3.7 с датой решения, согласовано с D5 — обоснованное улучшение. (2) Задача 2.4 «пересборка изменённого»: реализована полная пересборка в НОВЫЙ дашборд (4→5 на смоуке), in-place update отложен до живого update-пути адаптера — зафиксировано в CLAUDE.md (2.4). (3) Native filters по-прежнему deferred (F2/F3 Phase 1): превью честно предупреждает и в `spec_summary`, и в web UI — открытый вопрос владельцу вынесен в статус. Все три — приемлемые, документированные.
