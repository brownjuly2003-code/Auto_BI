## Резюме

Phase 4 закрывает весь backlog F1–F11 аудита Phase 3 (продуктизация DataLens + хардена). Эта сессия — два последних, стенд-зависимых остатка: **F6** (авторизованный healthcheck) и **полный F2** (rename-based атомарный ребилд). Обе правды реализованы аккуратно, **P1-нарушений инвариантов нет**, инварианты 1–8 выдержаны, тесты/линт зелёные. F6 — чистый и низкорисковый: `/ping` (liveness) дополнен дешёвым авторизованным `getWorkbookEntries scope=connection`, ошибка → `AdapterHealth(ok=False)` с понятным текстом; раньше «здоров» при мёртвой сессии. Симметрично закрывает F6-P3 из Phase 3.

F2 — самая рискованная правка (переработка live-верифицированного build-пути) — спроектирован верно по сути: build строит ВСЁ под temp-именами `<canonical>__wip`, canonical прошлой рабочей версии не трогается до полного успеха, и только потом `_promote_to_canonical` на каждый entry делает `delete(stale canonical) → renameEntry(temp→canonical)`. Это инвертирует прежнюю «delete-then-create на каждом шаге» (Phase 3 F2-P2): теперь сбой charts-engine посреди build пробрасывается с ЦЕЛОЙ прошлой версией (старый dataset/widget/dash живы, ничего canonical не удалено). Опорные допущения подтверждены живьём по CLAUDE.md: транспорт `us/renameEntry` (прямой REST `/v1/entries/:id/rename` не проксируется gateway = 404), **entryId стабилен при rename** → линки чарт↔дашборд (`chartId`=widget entryId) и URL `/{entryId}` валидны после промоута. Тесты грамотно инвертированы: `test_build_is_atomic_old_version_survives_mid_build_failure` пиннит `deletes==[]`/`renames==[]` при mid-build-сбое; точная gateway-последовательность зафиксирована (create-под-temp → promote-rename).

Остаточные слабости — **не баги happy-path, а窄кие окна частичного отказа ВНУТРИ `_promote_to_canonical`** и **отсутствие unit-теста на эти окна / на сам `_promote_to_canonical`** (вся «атомарность» доказана через build-уровень, но сама фаза промоута — delete-then-rename per-entry — последовательна и НЕ атомарна между собой; сбой между delete и rename, или после первого rename из трёх, оставляет смешанное canonical/temp-состояние). Самовосстановление на следующем build реально, но не для всех под-случаев одинаково чистое — детали в F1. Это сужает, но не отменяет выигрыш F2 (окно теперь — две быстрые US-операции без charts-engine вместо полного ребилда), и blast radius ограничен выделенным Auto_BI-workbook'ом (Phase 3 F3, уже закрыт). ## Findings

**F1 [P2] `auto_bi/adapters/datalens/adapter.py:488-506` (`_promote_to_canonical`) + `:510-553` (build) — фаза промоута не атомарна между entry, окно частичного отказа сужено но не закрыто; нет unit-покрытия самого промоута** — Атомарность F2 доказана на УРОВНЕ build (mid-build-сбой ⇒ promote не достигнут ⇒ прошлая версия цела — это верно и протестировано). Но САМ `_promote_to_canonical` — последовательный цикл из трёх независимых `delete→rename` пар (dataset, widget, dash), и между ними состояние смешанное. Разберём сбой ВНУТРИ промоута:
- (а) **Сбой между `delete(canonical)` и `renameEntry` одной пары** (docstring это признаёт, «self-heals on the next build»): остаётся canonical-дыра (старый удалён) + temp-сирота. На следующем build temp-сирота того же имени удалится temp-lookup'ом (`_delete_if_exists(_wip_name)` в create), новый canonical создастся промоутом — да, сходится. ✅ для этой пары.
- (б) **Сбой ПОСЛЕ rename dataset, но ДО rename widget/dash** (промоут прошёл частично, 1 из 3): теперь **новый dataset уже под canonical-именем и живёт**, а widget/dash — ещё под temp-именами `__wip`, СТАРЫЕ canonical widget/dash тоже ещё живут (их delete — позже в цикле, не достигнут). Дашборд (старый canonical) ссылается на СТАРЫЙ widget по entryId — но старый dataset УЖЕ удалён (его delete был перед rename нового) → **старый widget остался привязан к удалённому dataset-id** ⇒ старый дашборд может перестать рендерить до следующего успешного build. То есть окно «сломанного дашборда» сузилось (две US-операции вместо charts-engine-ребилда, как и заявлено), но НЕ исчезло полностью для случая частичного промоута. Это слабее, чем читается из ARCHITECTURE §3.5 («Rebuild АТОМАРЕН») и build-docstring («previous working version fully intact») — те утверждения строго верны только для сбоя ДО промоута, не ВНУТРИ него.
- (в) **Накопление `__wip`-сирот** при повторных провалах промоута: на следующий build temp-lookup чистит сироту ТОЛЬКО того же имени (`<canonical>__wip` детерминирован per title/chart.id), так что для одного и того же spec — да, перекрывается. Но если title/набор чартов между ретраями изменился (правка словами после неудачной сборки — штатный Phase-2 сценарий), старые `__wip`-сироты под прежними именами НЕ удаляются ничем и копятся в workbook. Прежний (delete-then-create) путь сирот не оставлял вовсе.

Почему важно: F2 продан как полная атомарность ребилда; на деле он переносит неатомарность из «дорогого» окна (charts-engine ребилд) в «дешёвое» (две US-операции) и для большинства транзиентных 5xx (которые бьют по charts-engine, т.е. ДО промоута) это реальный выигрыш. Но окно частичного промоута и `__wip`-накопление при сменах title остаются, а покрытия у `_promote_to_canonical` нет ВООБЩЕ (ни одного теста на сбой внутри промоута, на порядок delete-then-rename, на orphan-cleanup при смене имени). Все три build-теста проверяют либо полный успех, либо сбой ДО промоута.

Рекомендация: (1) **уточнить формулировки** ARCHITECTURE §3.5 / build-docstring: «атомарно на уровне build (промоут не достигается при сбое сборки); фаза промоута — последовательный per-entry delete-then-rename, узкое окно частичного состояния, самовосстановление на следующем build того же spec». (2) **тест на `_promote_to_canonical`**: смоделировать сбой `renameEntry` после первого успешного rename (FakeClient, бросающий на 2-м rename) и зафиксировать, что delete'ы/rename'ы выполнились в ожидаемом порядке и состояние предсказуемо. (3) **порядок промоута**: рассмотреть промоут dash ПЕРВЫМ или widget→dash до dataset-delete, чтобы старый дашборд не оставался ссылающимся на удалённый dataset (минимизировать (б)); либо удалять все stale canonical ПОСЛЕ всех rename'ов (rename temp→`<canonical>` коллизирует с живым canonical — значит сперва переименовать старый canonical в `__old`, потом rename temp, потом удалить `__old`; сложнее, но убирает окно). (4) **orphan-sweep**: на старте build опционально подчищать `*__wip`-сироты в scope (или задокументировать накопление при смене title как known-issue).

**F2 [P3] `auto_bi/adapters/datalens/adapter.py:296-303` (healthcheck) — авторизованный probe жёстко `scope="connection"`, не покрывает доступ к остальным scope** — F6-probe делает `getWorkbookEntries scope=connection`. Это подтверждает auth-cookie + форвард gateway + достижимость workbook (главная цель F6 — поймать мёртвую сессию до build). Но build пишет также в scope `dataset`/`widget`/`dash`, и теоретически доступ может различаться по scope (хотя в self-hosted OSS прав на workbook хватает на все entry-типы — латентно). Мелочь: probe — это «есть ли вообще доступ к workbook», и connection-scope для этого достаточен; гонять 4 lookup'а в healthcheck избыточно. Рекомендация: оставить как есть (одного scope достаточно для liveness+auth+reachability); максимум — комментарий, что connection-scope = репрезентативный probe доступа к workbook, не пер-scope ACL.

**F3 [P3] `auto_bi/adapters/datalens/adapter.py:541` — `chart_canonical = safe_entry_name(chart.title)` дублирует вычисление с `create_chart:413`** — В `build` canonical-имя чарта считается явно (`safe_entry_name(chart.title)`), чтобы положить в `to_promote`, и параллельно `create_chart` при `name=None` посчитал бы то же самое — но `build` передаёт `name=_wip_name(chart_canonical)`, так что внутренний расчёт не срабатывает. Если когда-нибудь `create_chart`/`ensure_dataset` начнут трансформировать имя иначе, чем `build` предполагает для canonical, промоут переименует temp в имя, которое не совпадёт с тем, что ищет idempotency-lookup. Сейчас корректно. Рекомендация: вынести «canonical-имя для chart/dataset/dash» в маленькие helper'ы (`_chart_canonical`/`_ds_canonical`), вызываемые и из `build`, и из публичных методов, чтобы единственный источник истины (мелочь, гигиена против будущего рассинхрона).

**F4 [P3] инвариант 7 / reversal §5.6-§5.7 — контрактный маркер `us/renameEntry` не добавлен в пин версии стенда** — F2 ввёл новую зависимость от gateway-экшена `us/renameEntry` (и от свойства «entryId стабилен при rename»).
