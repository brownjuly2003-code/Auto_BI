## Задача

Наполнить `GP_GOLDEN_CASES` (тот же `GoldenCase`, что и CH) под **GP-демо-схему**, прогнать
через живой GraceKelly, добиться порогов. ~10-14 кейсов: clear / ambiguous / infeasible
(+ при желании 1-2 итерации `edit`). Это НЕ копия CH-кейсов — схема другая (см. ниже).

## GP-демо-схема (`semantic/model_gp.yaml`) — ОТЛИЧАЕТСЯ от CH-демо

| Таблица | Колонки (роль) |
|---|---|
| `dm.sales` | `date`(time), `store_id`(dim→stores), `product_id`(dim→products), `revenue`(measure sum), `qty`(measure sum), `orders`(measure sum) |
| `dm.stores` | `store_id`, `city`, `format`, `region` |
| `dm.products` | `product_id`, `category`, `name` |

Joins (рёбра модели, для qualified-измерений): `sales.store_id→stores.store_id`,
`sales.product_id→products.product_id`.

**Чего НЕТ (в отличие от CH-демо `dm.sales_daily`):** `manager_id`, `items`, `price`,
`sales_daily`. Фактовая таблица называется `dm.sales` (не `sales_daily`). Меры: `revenue`,
`qty`, `orders`. Город — `dm.stores.city` (через join). Категория — `dm.products.category`.

## Как запускать (БД GP НЕ нужна)

Golden-сьют гоняет только агента (GROUNDING/CLARIFY/PROPOSE) и проверяет spec —
**дашборды не строит, EXPLAIN не зовёт** (Advisor в eval создаётся без `run_query`).
Нужны только: GraceKelly на 8011 + файл `model_gp.yaml`. Туннель к GP-стенду НЕ требуется.

```bash
# GraceKelly:
cd /d/GraceKelly && .venv/Scripts/python.exe -m uvicorn gracekelly.main:create_app \
  --factory --host 127.0.0.1 --port 8011
# eval (GP golden):
cd /d/Auto_BI && uv run auto_bi eval --model-path semantic/model_gp.yaml --suite golden
# точечный кейс: --cases gp_g1_revenue_by_day
```

## Кандидаты-кейсы (адаптировать/уточнить — это и есть S2-работа)

- **clear** (0 вопросов, проверяется spec):
  - `gp_g1` «Выручка по дням за июнь 2026» → table=`dm.sales`, cols `{date, revenue}`, viz line/area.
  - `gp_g2` «Общая выручка одним числом за июнь 2026» → `{revenue}`, big_number.
  - `gp_g3` «Топ-10 магазинов по выручке за июнь 2026» → `expect_columns_any=[{store_id, dm.stores.name}]`
    (с joins LLM законно берёт читаемое имя — но в GP-демо у stores нет `name`, есть `city`;
    решить: `store_id` либо `dm.stores.city`).
  - `gp_g4` «Топ городов по выручке за июнь 2026» → join, cols `{dm.stores.city, revenue}`, bar/table.
  - `gp_g5` «Число заказов по дням за июнь 2026» → `{date, orders}`.
  - `gp_g6` «Выручка по категориям товаров за июнь 2026» → join products, `{dm.products.category, revenue}`.
- **ambiguous** (вопрос обязателен): подобрать реальные двусмысленности GP-схемы. «Количество…»
  неоднозначно между `qty` и `orders` — хороший кандидат (`expect_phrase="количеств"`).
- **infeasible** (флаг, не галлюцинация): «Зарплата сотрудников…», «Возвраты…», «Погода…» —
  как в CH-сьюте (этих данных в DM нет).
