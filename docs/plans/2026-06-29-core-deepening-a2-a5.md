> (`ir/spec.py`) → по гейт-дисциплине проекта это **S4/владелец** («предложи варианты», как

## Что уже есть в контракте (grounded, не «из головы»)

- `Measure` (`ir/spec.py:96`): `column`, `agg`, `label`, опц. `transform: MeasureTransform`,
  опц. `denominator: Measure` (ratio). Производная мера компилируется в **двухуровневый**
  SELECT (`sqlgen._generate_windowed_sql:375`): inner GROUP BY базовых агрегатов → outer
  оконные функции / деление.
- `MeasureTransform` (`ir/spec.py:48`): `pop_abs`/`pop_pct`/`yoy_pct`/`share_of_total`/
  `running_total`.
- `TimeGrain` (`ir/spec.py:74`): day/week/month/quarter/year.
- `Viz` (`ir/spec.py:28`): big_number/line/bar/stacked_bar/area/pie/table/pivot/heatmap.
- **Механика лага-на-k УЖЕ существует** (`sqlgen._window_expr:356` `lag(k)` + `_periods_per_year`
  + `yoy_lag`): yoy лагает на `_periods_per_year(grain)` строк через `exp.Lag(offset=k)` с явным
  ROWS-фреймом. Параметризовать k извне — почти бесплатно.
- Валидация (`ir/validate.py`): `_ORDERED_TRANSFORMS` (`:26`) требуют первое измерение = TIME;
  `_TRANSFORM_UNSUPPORTED_VIZ` (`:35`) = {big_number, pivot, heatmap}; yoy требует не-day grain
  (`:241`).
- **Демо-модель** (`semantic/model.yaml`): `dm.sales_daily(date, store_id, product_id,
  manager_id, revenue, orders, items)` — грейн date×store×product, БЕЗ клиентской сущности;
  `dm.products(id, name, category, brand, price)` — `price` = сырая мера (2000 строк).

## A2 — Лаг на произвольные N периодов (обобщение yoy)

**Что.** «Изменение vs N периодов назад» (vs 3 месяца назад, vs 7 дней назад) — обобщение yoy,
где сейчас лаг жёстко = периодов-в-году.

**Вариант A2-v1 (РЕКОМЕНДУЮ) — опц. поле `lag_periods: int | None` на мере.**
- Контракт: добавить `Measure.lag_periods: int | None = None`. Работает с `pop_abs`/`pop_pct`
  (и при желании отдельно): «pop_pct + lag_periods=3» = % к значению 3 периода назад. `None` →
  текущее поведение (k=1), `yoy_pct` игнорирует поле (k=периодов-в-году).
- SQL: в `_window_expr` маршрутизировать `k = measure.lag_periods or 1` в существующий `lag(k)`.
  **0 новой SQL-машинерии.**
- Валидация: `lag_periods` (если задан) ≥1; разрешён только с `pop_abs`/`pop_pct`; те же
  ordered-правила (первое измерение = TIME). `model_dump` идемпотентен (новое опц. поле).
- Alias: `measure_alias` дополнить суффиксом `_lag<N>` чтобы не коллидить (`pop_pct_..._lag3`).
- Verify: DuckDB (PG-семантика) + CH live (дисциплина проекта для числовых правок).
- **Оценка: S. Риск: низкий** (механика есть, не-задействованный путь байт-идентичен).

**Вариант A2-v2 — новые transform `lag_abs`/`lag_pct` + параметр.** Чище семантически, но дублирует
pop_* и расширяет enum на 2 значения + промпт/адаптеры. Не рекомендую (pop_* + параметр выразит то же).

## A3 — Кумулятивная доля / Pareto (running-share, ABC-анализ)

**Что.** Категории отсортированы по мере убыв., накопленная доля → «топ-3 категории = 80%
оборота» (классический ABC/Парето). Реальный, частый аналитический паттерн.

**Ключевое отличие от существующего.** `running_total` упорядочен по ВРЕМЕНИ; `share_of_total` —
доля от всего без порядка. Pareto = `SUM(agg) OVER (ORDER BY agg DESC ROWS UNBOUNDED PRECEDING) /
SUM(agg) OVER ()` — окно упорядочено по **МЕРЕ убыв.**, не по времени. Это **ломает инвариант**
`_ORDERED_TRANSFORMS требуют TIME` (`validate.py:26,232`).

**Вариант A3-v1 (РЕКОМЕНДУЮ) — новый transform `running_share`, окно по мере.**
- Контракт: `MeasureTransform.RUNNING_SHARE`. Семантика: «накопленная доля по убыванию меры».
- SQL: в `_window_expr` ветка — окно сортируется по `src DESC` (не по `order_col`): кумулятивная
  сумма / total. Требует ≥1 категориальное измерение (как share_of_total), НЕ требует TIME.
- Валидация: `running_share` НЕ в `_ORDERED_TRANSFORMS` (отдельный список); требует ≥1 измерение;
  запрещён на big_number/pivot/heatmap. is_percent_measure → True (рендер %).
- Verify: DuckDB + CH live (кумул-доля категорий должна монотонно расти до 1.0).
- **Оценка: M. Риск: средний** (новая семантика порядка окна — но изолирована одной веткой).

**Вариант A3-v2 — композиция в autospec без нового transform.** Невозможно чисто: нет примитива
«окно по мере». Отвергнут.

## A4 — Гистограмма / распределение (бакетинг меры)

**Что.** Распределение значений по корзинам: «сколько товаров в каждом ценовом диапазоне»,
«распределение дневной выручки магазинов». Демо-кандидат: `dm.products.price` (2000 строк).

**Ключевое отличие.** Не GROUP BY измерения, а GROUP BY **вычисляемого ведра** сырой/агрег.
колонки: `floor((x - min)/width)` → COUNT. Новый `Viz` + bin-логика + оба адаптера.

**Вариант A4-v1 — гистограмма сырой колонки.** Бинит сырую meas-колонку (price) по фактовым
строкам. На 20M строк CH потянет, но семантика «распределение по строкам факта» доменно-зыбкая
(строки sales_daily — это дни×магазины×товары, не «наблюдения»).

**Вариант A4-v2 (РЕКОМЕНДУЮ если брать A4) — гистограмма агрегата.** Сначала суб-агрегат по
сущности (выручка per store), затем бакетинг — «распределение магазинов по обороту». Семантически
чисто и универсально.
- Контракт: `Viz.HISTOGRAM` + параметры бинов (`bins: int` или `bin_width`) — нужно новое поле
  на ChartQuery или ChartSpec.
- SQL: трёхуровневый (агрегат по сущности → ведро → COUNT) ИЛИ двухуровневый для сырой колонки.
- Адаптеры: Superset имеет нативную histogram-визу; DataLens — bar по подписям-вёдрам (как
  heatmap→stacked-bar деградация, память `autobi-bi-engine-limits`).
- Валидация: новая viz-shape ветка; autospec-интеграция опц.
- Verify: оба BI на стенде (render-verify новой визы) — **частично стенд-gated**.
- **Оценка: L. Риск: средний-высокий** (наибольшая контракт-поверхность: Viz + bins + 2 адаптера
  + validate + возможно autospec; DataLens-рендер требует стенда).

## A5 — Когортный / retention анализ

**Что.** «Из клиентов, впервые пришедших в январе, сколько вернулось через 1/2/3 месяца» —
retention-матрица.

**🔴 Блокер фезибилити (проверено по `semantic/model.yaml`).** Когорта требует **сущностной
идентичности**: entity, появляющийся впервые в период T, отслеживаемый на возврат. В демо-модели
такой сущности НЕТ — `sales_daily` = дневные агрегаты по магазин×товар, не транзакции с клиентом.
store_id/product_id «существуют всегда» (справочники), не «появляются и возвращаются».

**Последствия.**
- Невозможно verify на текущем демо (нет данных для когорт).
- Высокий риск нарушить `universal-not-retail`: retention доменно-специфичен (клиенты/подписки),
  «зашить» его в универсальное ядро — ровно то, от чего предостерегает память.
- Самый большой контракт (новый модуль анализа, новая spec-форма: cohort-key + period-offset +
  retention-measure).

**Вариант A5-v1 — отложить (РЕКОМЕНДУЮ).** Брать только если (а) появится модель с клиентской
сущностью и (б) владелец подтвердит, что retention входит в scope универсального продукта.
Иначе = переусложнение под несуществующий кейс.
