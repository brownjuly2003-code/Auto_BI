## 1. Транспорт (подтверждает вывод спайка 3.1)

Адаптер ходит через **UI-gateway** `POST http://<datalens-ui>:8080/gateway/root/<svc>/<method>`
(JSON, **cookie-сессия `auth`**). Это deployment-реалистично: в self-hosted деплое UI — единственная
публичная точка, бэкенды (control-api/us/auth/data-api) внутренние. Gateway сам расшифровывает
cookie и подмешивает `Authorization: Bearer <JWS>` бэкендам.

**Важно:** gateway корректно форвардит тело для `bi/createConnection` и `bi/createDataset`
(✅ оба отработали). Баг тела изолирован в `bi/validateDataset` (роняет тело → 415, см. 3.1) —
**мы его НЕ используем**: схему датасета даём детерминированно из IR (см. §4), валидация не нужна.

## 2. Auth ✅ (signin → cookie `auth`)

Auth-сервис (`@gravity-ui/expresskit`, Node) маршруты (из `dist/server/routes.js`):
`POST /signin` (passport local), `POST /refresh`, `GET /logout`, `POST /signup`,
`POST /private/v1/management/users/*` (master-token).

**Login ✅:** `POST {auth}/signin` JSON `{login:"admin", password:"admin"}` → `200 {"done":true}`,
ставит cookie **`auth`** (+ `auth_exp`). `auth` = **JWE (5 частей, encrypted)** — НЕ plain JWS;
control-api напрямую её не принимает («Invalid token»). Gateway расшифровывает её сам.

**Следствие для адаптера:** держать cookie-jar; `auth` cookie приложить ко всем `/gateway/root/*`
вызовам. (Для прямого control-api нужен был бы JWS, который минтит gateway — не пошли этим путём.)

control-api env (для справки): `AUTH__TYPE=NATIVE`, JWT `PS256` (`AUTH__JWT_KEY` = public). US:
`AUTH_ENABLED=true`, `MASTER_TOKEN=us-master-token` (US-REST CRUD по master-token ✅ — `DELETE
/private/entries/{id}` с `x-us-master-token` → 200; пригодится для chart/dashboard US-entry).

## 3. Connection ✅ (`bi/createConnection`)

`POST /gateway/root/bi/createConnection` (cookie) →
```json
{ "name": "...", "workbook_id": "<wb>", "type": "clickhouse",   // GP: "greenplum"
  "host": "host.docker.internal", "port": 8123, "username": "auto_bi_ro",
  "password": "...", "secure": "off",        // secure:"on"=HTTPS; CH :8123 plain → "off"
  "raw_sql_level": "subselect",              // ОБЯЗАТЕЛЬНО для dataset-from-SQL
  "cache_ttl_sec": null }
```
→ `200 {"id":"<connectionId-str>"}`. **Поле типа на входе = `type`** (НЕ `db_type`; `db_type`
только в ОТВЕТЕ `getConnection`). CH-host из BI = `host.docker.internal` (DataLens и
`auto_bi_clickhouse` в разных docker-сетях). `bi/createConnection` бэкенд = control-api
`POST /api/v1/connections`.

## 4. Dataset ✅ (`bi/createDataset`, схема из IR — без validate!)

`POST /gateway/root/bi/createDataset` (cookie) →
```json
{ "workbookId": "<wb>", "name": "auto_bi__<...>",
  "dataset": {
    "sources": [{
      "id": "<uuid>", "title": "<src>", "connection_id": "<connId>",
      "source_type": "PG_SUBSELECT",                 // CH: "CH_SUBSELECT"
      "parameters": { "subsql": "<generate_chart_sql(query)>" },
      "managed_by": "user",
      "raw_schema": [                                 // ← КЛЮЧ: колонки subselect из IR
        { "name": "category", "title": "category", "user_type": "string",
          "native_type": null, "nullable": true },
        { "name": "amount", "title": "amount", "user_type": "integer",
          "native_type": null, "nullable": true }
      ],
      "index_info_set": []
    }],
    "source_avatars": [{ "id":"<uuid>", "source_id":"<source.id>", "title":"<src>",
                         "is_root": true, "managed_by": "user" }],
    "avatar_relations": [],
    "result_schema": [                                // поля = колонки (из IR)
      { "guid":"<uuid>", "title":"category", "source":"category", "data_type":"string",
        "cast":"string", "type":"DIMENSION", "aggregation":"none", "calc_mode":"direct",
        "avatar_id":"<avatar.id>", "hidden":false, "managed_by":"user",
        "description":"", "formula":"", "valid":true }, ...
    ],
    "rls": {}, "component_errors": { "items": [] }
  } }
```
→ **`200`** ✅ (вернул dataset c `options.supported_functions` и т.д.; есть `id`).

**Почему без validate:** control-api `DatasetCreate` при создании нового источника читает
`source["raw_schema"]` (+ `index_info_set`) — если их нет, `KeyError: 'raw_schema'` → 500. Обычно
их выводит `validateDataset` (интроспектит БД). **Но наш адаптер знает колонки и роли из IR**
(`ChartQuery`: dimensions/measures/series/...), поэтому строит `raw_schema`+`result_schema`
детерминированно — это даже точнее инварианту #1 (детерминированный код адаптера, БД-introspection
не нужна). Маппинг IR-тип → `user_type` (DataLens `UserDataType` enum): string/integer/float/date/
datetime/boolean. `native_type:null` принят. Поля источника обязательны: `raw_schema`,
`index_info_set` (=[]), `parameters`, `managed_by`, `source_type`; у avatar — `managed_by`,
`is_root`. `raw_schema` item-схема = `RawSchemaColumnSchema` (`name,title,user_type,native_type?,
nullable,description?,has_auto_aggregation?,lock_aggregation?`).

**Идемпотентность:** как в Superset (имя `auto_bi__<slug>__<hash>`), для update —
`bi/updateDataset` (метод не реверсил; на 3.2 проверить).

### 5.1 ✅ Login-эндпоинт ЗАПИНЕН (live 200 + cookie)

Путь = **`POST /gateway/auth/auth/signin`** (scope буквально `auth`, НЕ `root`). Тело
`{login,password}` → `200 {"done":true}` + `Set-Cookie: auth=…` (URL-enc JSON
`{accessToken,refreshToken}` с JWS PS256; gateway сам её принимает на `/gateway/root/*`).
Причина прежнего 401 на `/gateway/root/auth/signin`: scope `root` — auth-гейтед (циркулярно);
auth обслуживает выделенный роут `POST /gateway/:scope(auth)/:service/:action` c
`AuthPolicy.disabled` (`dist/server/components/auth/routes.js`). **Зашито в `client.py`
`DEFAULT_SIGNIN_PATH` (закрыт §5.1).** Транспорт RPC = `/gateway/root/<svc>/<method>` (cookie).

### 5.2 ✅ Chart (US widget-entry) `data`-блоб — РЕВЕРСНУТ

Структура entry.data = `{js, ui, url, graph, table, params, shared}`:
- **6 стабов** (js/ui/url/graph/table/params) — boilerplate, зовут `require('libs/datalens/v3')`
  с `ChartEditor.getSharedData()`/`getParams()`. Под нашим контролем → **генерить НАЧИСТО**
  (demo-стабы хранят искажённый плейсхолдер `$99` вместо тернарного `?` в `graph`-стабе — НЕ копировать).
  Стаб-провайдер по node-type: graph→`buildHighchartsConfig`(в `graph`), table→`buildChartsConfig`(в `table`),
  metric→как graph. `js`=buildGraph, `url`=buildSources, `ui`=buildUI, `params`=buildParams.
- **`shared`** (JSON-строка) — реальный конфиг: `{datasetsIds:[dsId], datasetsPartialFields:[[{guid,title,calc_mode}…]],
  visualization:{id,type,placeholders:[…]}, colors:[], sort:[], labels:[], filters:[], extraSettings:{…},
  type:"datalens", version:"4", updates:[], geopointsConfig:{}, hierarchies:[], shapes:[], …}`.

**Маппинг IR `Viz` → `visualization.id` (node) + раскладка placeholders** (снято live):

| IR Viz | viz_id | node | placeholders (роль→секция) |
|---|---|---|---|
| `big_number` | `metric` | metric_wizard_node | `measures`=[measures[0]] |
| `line` | `line` | graph_wizard_node | `x`=[dim0], `y`=measures, (`y2`=[]); series→`shared.colors` |
| `bar` | `column` | graph_wizard_node | `x`=[dim0], `y`=measures; series→`colors` |
| `stacked_bar` | `column` | graph_wizard_node | как bar + стек (наличие `colors`/неск. мер; 100%=`column100p`) |
| `area` | `area` | graph_wizard_node | `x`=[dim0], `y`=measures; series→`colors` |
| `pie` | `pie` | graph_wizard_node | `dimensions`=[dim0], `measures`=[measures[0]]; `sort`+`labels`=мера |
| `table` | `flatTable` | table_wizard_node | `flat-table-columns`=group_columns()+measures |
| `pivot` | `pivotTable` | table_wizard_node | `rows`=rows, `pivot-table-columns`=columns, `measures`=measures |
| `heatmap` | **нет в Wizard** | — | **деградация → `pivotTable`** (rows=[dim0],columns=[dim1],measures=[m0]) + пометка build-log (инвариант §3.4); нативный декартов heatmap только Editor/HC-путь |

**field-объект item** (в placeholders.items[] и colors/sort/labels) — берётся из dataset
result_schema, форма: `{type:DIMENSION|MEASURE, calc_mode:"direct", data_type, initial_data_type,
cast, aggregation, source:<alias>, guid:<field guid>, title, avatar_id:<dataset avatar>,
datasetId, datasetName, id:"dimension-N"/"measure-N", managed_by:"user", valid:true, hidden:false,
formula:"", description:"", autoaggregated:false, virtual:false, lock_aggregation:false,
aggregation_locked:false, has_auto_aggregation:false, guid_formula:"", default_value:null,
value_constraint:null}`. Биндинг к датасету — по `guid`+`avatar_id` из createDataset-ответа.
**РИСК (S5):** `visualization` в demo несёт ещё `name/icon/allowFilters/placeholders[].settings/
allowedTypes` — минимальный принимаемый набор полей проверить live-рендером (как form_data в Superset).

### 5.3 ✅ Dashboard (US dash-entry) `data`-блоб — РЕВЕРСНУТ

`{salt:<str>, schemeVersion:7, counter:<int>, tabs:[…], settings:{…}, description}`. Tab =
`{id, title, items:[…], layout:[…], aliases:{default:[[…]]}, connections:[…]}`:
- **`items[]`**: виджеты/текст/селекторы — `{id, type:"widget"|"text"|"control", data:{…}, namespace:"default"}`
  (widget.data ссылается на chartId; text.data.text=markdown).
- **`layout[]`**: грид-плейсмент `{i:<itemId>, x, y, w, h}` (по аналогии с position_json Superset; w в колонках).
- **`connections[]`** + **`aliases`**: связи селектор→чарт (scope-to-applicable, prep §A.3) —
  `{from:<selectorId>, to:<itemId>, kind:"ignore"|…}`, alias = пара `[fieldGuid, paramName]`.
- `settings`: `{autoupdateInterval, dependentSelectors:true, …}`.
Создаётся US-entry `scope:"dash"` master-token CRUD. Селекторы (`spec.filters`) переносят
scope-to-applicable из `participating_chart_ids` (native_filters.py:54).
