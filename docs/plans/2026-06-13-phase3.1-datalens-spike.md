# Phase 3.1 — DataLens spike (reverse API на self-hosted инстансе)

Дата: 2026-06-13. Инстанс: self-hosted open-source DataLens на Mac (`~/datalens_dl`,
`HC=1`, gateway-версия `4.10.4`, `APP_INSTALLATION=opensource`), UI :8080, туннель Windows
`ssh -N -L 8090:localhost:8080 deproject-mac`, логин admin/admin. Метод реверса — драйв живого
UI + захват сети (Playwright) + прямые gateway-вызовы из аутентифицированной страницы (`fetch`,
cookie-auth) + чтение логов backend-контейнеров на Mac. Инвариант проекта соблюдён: форматы
реверсятся с живого стенда, ключевые вызовы подтверждены реальными ответами (не угаданы).

## 0. Вердикт: **GO (условный)** — адаптер реализуем, но НЕ через UI-gateway

DataLens-адаптер (3.2) реализуем на self-hosted инстансе. **Ключевой пересмотр архитектуры
относительно prep §A.0:** адаптер должен бить в **REST бэкендов напрямую** (control-api + US),
как Superset-адаптер бьёт в Superset REST, а **НЕ** через UI-gateway `/gateway/root/<svc>/<method>`.
Причина — баг форвардинга тела в gateway-экшене `validateDataset` (см. §3). Программный путь
создания всех сущностей подтверждён; Highcharts включён. Остаток (точные config-блобы chart/dashboard)
— механический реверс на 3.2 против живого стенда по тому же контракт-тест-паттерну, что Phase 0–2.

Также §A.0 переформулируется в выгодную сторону: на self-hosted **чарты создаются как US-entry
напрямую** (а не через облачный `createEditorChart` Public API), поэтому ограничение «Wizard через
Public API не создаётся» здесь **не действует** — адаптер может писать в US хоть Wizard-, хоть
Editor-конфиг (что богаче выберем).

## 2. Транспорт и auth (швы адаптера)

Два слоя:

| Слой | Что это | Роль для адаптера |
|---|---|---|
| **UI-gateway** `POST /gateway/root/<svc>/<method>` (JSON, **cookie-сессия**) | BFF Node-сервиса `datalens-ui`; проксирует на бэкенды, подмешивает `Authorization: Bearer <JWT>` | **НЕ использовать** для записи (баг `validateDataset`, §3). Годится только как способ реверса/чтения с браузера |
| **Бэкенды REST напрямую** | `control-api` (Flask, `/api/v1/...`) для connection/dataset; `us` (united-storage, Node, `/v1/...`+`/private/...`) для chart/dashboard-entry; `data-api` для данных/preview | **Цель адаптера** (как Superset REST) |

**Auth бэкендов (env):**
- `control-api`: `AUTH__TYPE=NATIVE`, JWT `PS256` (есть `AUTH__JWT_KEY`). Прямой вызов без токена →
  `401 {"message":"User access token header is missing"}`. ⇒ адаптеру нужен **user-JWT** (выдаёт
  сервис `datalens-auth`, NATIVE; логин admin/admin). Это и есть «auth-шов» из runbook.
- `us`: `AUTH_ENABLED=true`, `MASTER_TOKEN=us-master-token`. **US-REST CRUD по master-token
  подтверждён живьём**: `DELETE /private/entries/{id}` с заголовком `x-us-master-token: us-master-token`
  → `200` (см. §6, уборка). ⇒ chart/dashboard (US-entry) можно создавать/удалять US-мастер-токеном
  напрямую.

## 3. КРИТИЧНО — баг UI-gateway `validateDataset` (Content-Length 0 / form-urlencoded)

Симптом: `POST /gateway/root/bi/validateDataset` → **415 Unsupported Media Type** и для моего
`fetch`, и **для самого UI** (диалог добавления SQL-источника не закрывается — создание датасета
в UI заблокировано).

Root cause (из лога `datalens-control-api`): gateway форвардит на бэкенд
`POST /api/v1/datasets/validators/dataset` с **`Content-Type: application/x-www-form-urlencoded` и
`Content-Length: 0`** — тело JSON **потеряно**. Для сравнения, `ConnectionCreate` пришёл на
control-api корректно (`application/json`, `Content-Length: 226`) и отработал.

Доказательство, что это **только баг gateway**, а не ограничение API: прямой вызов
`control-api` `POST /api/v1/datasets/validators/dataset` с корректным `Content-Type: application/json`
(минуя gateway) вернул **`401` (нужен auth), а НЕ `415`** ⇒ эндпоинт принимает тело, проблема
исключительно в gateway-прокси версии 4.10.4.

**Следствие:** адаптер 3.2 **обязан** ходить в control-api/US REST напрямую с
`Content-Type: application/json` + JWT/master-token. Это и так предпочтительный паттерн (серверный
адаптер, как Superset). UI-gateway для записи не использовать.

## 4. Реверс API по сущностям

### 4.1 Connection — ✅ ПОДТВЕРЖДЕНО ЖИВЬЁМ (программно создан)

- UI-gateway: `bi/createConnection` (для справки). Прямой live-вызов из страницы:
  `bi/createConnection {name, workbook_id, type:"clickhouse", host, port, username, password,
  secure, raw_sql_level:"subselect", cache_ttl_sec}` → **`200 {id:"<str>"}`** (создано две CH-связи,
  затем удалены).
- Бэкенд-эквивалент адаптера: **`POST /api/v1/connections`** (control-api), `endpoint_code=ConnectionCreate`.
- Поле типа на ВХОДЕ — `type` (не `db_type`!). На ВЫХОДЕ `getConnection`/`GET /api/v1/connections/{id}`
  отдаёт `db_type`, `host`, `port`, `username`, `secure`("on"/"off"), `raw_sql_level`, `cache_ttl_sec`,
  `workbook_id`, `options{allow_dataset_usage, allow_typed_query_usage, query_types[], allow_dashsql_usage}`.
- **`raw_sql_level:"subselect"`** включает SQL-subselect-источники (= наш «один валидированный-SQL
  датасет на чарт»). Без него — `disabled`. Для CH `type:"clickhouse"`, для GP — `type:"greenplum"`.
- Прим.: `secure:"on"` ⇒ HTTPS; для CH :8123 (plain HTTP) ставить `secure:"off"`. Хост из BI-сети —
  `host.docker.internal` (DataLens и `auto_bi_clickhouse` в РАЗНЫХ docker-сетях; container-имя
  `clickhouse` из `.env AUTO_BI_CH_HOST_FROM_BI` НЕ резолвится из DataLens-стека).

### 4.2 Источники коннекшна (для dataset-from-SQL)

`GET /api/v1/connections/{id}/info/sources` (UI-gateway `bi/getConnectionSources`) → `freeform_sources`
содержит дескриптор subselect:
- PG: `{source_type:"PG_SUBSELECT", connection_id, parameters:{subsql:"<SQL>"}, form:[{name:"subsql",
  input_type:"sql", template_enabled:true}]}`. Для CH аналогично `CH_SUBSELECT`. Плюс `PG_TABLE`/`CH_TABLE`
  с `{schema_name, table_name}`.

### 4.3 Dataset — структура реверснута; create через control-api direct (3.2)

- **Модель датасета** (из `us/getEntry` demo `Orders`, поле `data`): `result_schema[]`
  (поля: `guid,title,type:DIMENSION|MEASURE,cast,data_type,aggregation,calc_mode,formula,avatar_id,source`),
  `source_avatars[]` (`{id, source_id, title, is_root}`), `avatar_relations[]`, `obligatory_filters`, `rls`.
- **Validate-эндпоинт:** `POST /api/v1/datasets/validators/dataset` body
  `{dataset:{sources,source_avatars,avatar_relations,result_schema,rls,component_errors}, updates:[
  {action:"add_source", source:{id,title,connection_id,source_type:"PG_SUBSELECT",parameters:{subsql}}},
  {action:"add_source_avatar", source_avatar:{id,source_id,title}} ]}` → возвращает датасет с
  авто-выведенным `result_schema` (типы колонок из БД). **Этот эндпоинт рабочий при прямом вызове
  с JSON** (см. §3; через gateway — 415). Формат `updates` подтверждён захватом тела, что слал UI.
- **Create:** `POST /api/v1/datasets` (`workbook_id` + валидированный `dataset`) → entry в US.
  Точный live-прогон требует user-JWT (см. §2) — финализируется на 3.2.

### 4.4 Chart (widget) — US-entry; программный путь подтверждён, блоб реверсится на 3.2

- Чарт = US-entry `scope:"widget"`. Создаётся записью в US (`us/createEntry` / `POST /v1/entries`)
  с `data`-блобом конфига чарта. **На self-hosted доступен любой тип блоба** (Wizard `graph_wizard_node`
  или Editor/Highcharts) — облачное ограничение «только createEditorChart» (prep §A.0) НЕ применяется.
- Программная достижимость US-CRUD подтверждена (§2, master-token DELETE → 200). **Точный блоб
  (биндинг IR viz → Wizard-секции / Highcharts-серия + ссылка на dataset)** не реверснут — в demo нет
  готового чарта; цель реверса 3.2: создать 1 чарт через Wizard на живом датасете и снять `data`
  (как form_data снимали с Superset).
- Маппинг 9 viz IR → DataLens из prep §A.2 остаётся валидной справкой (line/bar→column/pie/area/
  table/pivot/big_number→indicator/heatmap-нативный-при-HC).

### 4.5 Dashboard — US-entry `scope:"dash"`; путь тот же

- Дашборд = US-entry `scope:"dash"` (`us/getWorkbookEntries scope:"dash"`). Создаётся записью в US
  с блобом `{tabs:[{items:[widget|selector...]}]}`. Селекторы (фильтры) — нативный механизм
  (prep §A.3 scope-to-applicable переносится). Блоб реверсится на 3.2 (создать дашборд с чартом, снять).

## 5. Влияние на план 3.2 (что стало механическим, что осталось)

**Механически готово (есть метод+шейп, часть live-подтверждена):**
- Connection (CH/GP) → control-api `POST /api/v1/connections`, `type:"clickhouse"|"greenplum"`,
  `raw_sql_level:"subselect"`. ✅ live.
- Dataset → validate (`/datasets/validators/dataset`) + create (`/datasets`) с subselect-источником
  (`{PG|CH}_SUBSELECT`, `parameters.subsql` = наш `generate_chart_sql`). Шейп + validate-формат сняты.
- Транспорт/auth: control-api (user-JWT, NATIVE admin/admin) + US (master-token). Минуя gateway.

**Остаток реверса на 3.2 (против живого стенда, контракт-тесты):**
1. Получение user-JWT из `datalens-auth` (NATIVE-login) — auth-клиент адаптера.
2. Точный `data`-блоб **chart** (Wizard или Highcharts-Editor) — создать 1 через UI, снять, шаблонизировать.
3. Точный `data`-блоб **dashboard** (+ селекторы) — аналогично.
4. End-to-end: connection→dataset→chart→dashboard программно + рендер с реальными данными (CH-стенд).

**Архитектура адаптера (`adapters/datalens/`):** `client.py` (auth-JWT + control-api/US REST,
`Content-Type: application/json`), `dataset.py` (IR SQL → subselect-датасет), `chart_config.py`
(IR viz → Highcharts/Wizard блоб), `adapter.py` (5 методов Protocol). Шов ref `id:int|str` (S4-2) готов.
