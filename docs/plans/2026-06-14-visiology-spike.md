# Visiology — спайк адаптера: вердикт NO-GO (автономно), gate на лицензионный стенд

Дата: 2026-06-14. Часть остатка Phase 4 (продуктовые опции: Visiology/Luxms-адаптеры).
Источник — веб-research по публичным докам (`visiology-doc.atlassian.net`, `ru.visiology.su`,
community-wrapper `github.com/codingjerk/visiology-py`). Парная оценка Luxms — в
`2026-06-14-luxms-adapter-plan.md`.

## TL;DR

**Visiology НЕ подходит для автономного адаптера.** Архитектурно это **import + visual-designer**
платформа (ближе к Power BI, чем к Superset/DataLens): дашборды собираются **только в визуальном
Designer**, у авторинга (наборы данных / виджеты / дашборды) **нет публичного REST API**. Это
ломает саму модель Auto_BI «детерминированный адаптер строит дашборд через API». Плюс нет
free/email-only стенда для реверса. **Вердикт: NO-GO автономно; gate на лицензионный стенд v3
от заказчика.**

## Что есть в API (и чего нет)

| Этап адаптера | Visiology 3.x | Годен для адаптера? |
|---|---|---|
| Connection к внешней БД | через UI-форму (JDBC); есть ClickHouse/Greenplum/PG/MySQL/MSSQL/Oracle | нет публичного create-API |
| Dataset из SQL | UI: выбрать источник → `SELECT` → «Загрузить»; данные **материализуются** в ViQube (не live-query) | нет; модель — импорт, не subselect |
| Charts/widgets | **только** в Designer (drag полей на оси) | **нет REST** (JS API лишь конфигурит уже созданные виджеты) |
| Dashboard | только Designer | **нет REST** |
| Data **loading** (refresh) | **есть** REST: `POST /viqube/loaders`, `/loaders/{id}/execute`, dataset-refresh (v3.14) | да, но это про подачу данных, не про авторинг |

**Auth** (не блокер): username/password → bearer-токен (ViQube ApiV2/v3), `X-API-VERSION`. Community
wrapper `visiology-py` оборачивает именно loader/data-API.

**Config-as-code:** наборы данных можно экспортнуть/импортнуть JSON-файлом, но это **ручная**
UI-выгрузка, переносит только структуру (без данных и без дашбордов), с переподключением таблиц.
Программного import-эндпоинта в публичных доках нет.

## Два независимых блокера

1. **Нет programmatic dashboard-authoring API.** Авторинг — UI-only Designer; единственный
   реверс-путь = браузерная автоматизация Designer'а или реверс недокументированных внутренних
   эндпоинтов на лицензионном инстансе. Хрупко и вне скоупа детерминированного адаптера.
2. **Нет free/email-only стенда.** Дистрибутивы — по `license# + корпоративная почта`; публичный
   demo только v2 (стирается ежедневно), v3-trial — по заявке (sales-gated). Реверс/верификация
   автономно невозможны.

> Оговорка по уверенности: «нет authoring-API» — по публичным докам. Внутренние эндпоинты Designer
> могут существовать, но не задокументированы — это «не найдено в публичных доках», а не «доказано
> отсутствие». Их реверс всё равно требует лицензионного стенда.

## Решение и gate

- **NE реализуем** Visiology-адаптер автономно и **NE добавляем** `TargetBI.VISIOLOGY` (enum-значение
  без рабочего адаптера = мёртвый путь, нарушил бы инвариант «один spec → N реальных BI»).
- **Gate (нужно от владельца/заказчика):** лицензионный стенд Visiology 3.x + доступ. Тогда —
  спайк-реверс (есть ли пригодный внутренний authoring-эндпоинт; иначе оценка browser-driving
  Designer). Только после этого решать GO/NO-GO на реализацию.
- Рекомендация по приоритету: **Luxms приоритетнее** (полный REST/CRUD, живой публичный API,
  ложится на существующий shape адаптеров) — см. `2026-06-14-luxms-adapter-plan.md`.
