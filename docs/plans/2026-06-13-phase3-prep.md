## A. IR → DataLens capability matrix (задача 3.2)

Источники: DataLens Public API (`https://api.datalens.tech`, OpenAPI), visualization-ref
(`yandex-cloud/docs`), концепты Charts/Wizard. Грунт под §A — реальные доки, не догадки; точные
имена секций Wizard и наличие «dataset from SQL» в Public API **подтверждаются спайком 3.1** (помечено TBD).

### A.0. КРИТИЧНО (находка 2026-06-13): Public API создаёт **Editor-чарты, не Wizard**

Полный список `create*`-методов Public API (с портала `api.datalens.tech`): `createConnection`,
`createDataset`, `createDashboard` [experimental], **`createEditorChart` [experimental]**. **Метода для
Wizard-чартов (drag&drop-визуализатор) НЕТ.** Значит:

- **Wizard-чарты программно не создаются** — раскладка ролей по секциям Wizard (§A.2 ниже) остаётся только
  справкой по UI-семантике, НЕ целью адаптера.
- **Цель DataLens-адаптера — Editor-чарты** (`createEditorChart`): ChartEditor компилирует чарт из конфига
  (на инстансах с Highcharts, `HC=1` — Highcharts-конфиг; иначе DataLens-native graph). Чарт читает датасет
  (наш «dataset from SQL» сохраняется) и рендерит по сгенерированному конфигу. Это **другой компиляционный
  таргет**, чем Superset form_data: IR viz → graph-config (тип серии + биндинг полей), а не Wizard-секции.
- **Следствие для heatmap:** у Highcharts есть **нативный декартов heatmap** (серия `heatmap`, точки `[x,y,value]`)
  → на Editor-пути **деградация heatmap из §A.2 СНИМАЕТСЯ** (при включённом HC). Без HC — деградация в native-аналог
  остаётся (TBD спайк).

**Go/no-go спайка 3.1 переформулирован:** (1) работает ли `createEditorChart` через Public API на целевом
инстансе; (2) реверс payload-shape editor-чарта (как form_data реверсили с Superset); (3) **доступен ли
Highcharts (`HC=1`)** — от этого зависят доступные типы серий и heatmap. No-go (editor-чарты недоступны / HC нет
и native беднее) → остаёмся Superset-only до спроса (заложено в PLAN exit-criteria Phase 3).

### A.1 Модель DataLens vs Superset (что чему соответствует)

| Шаг адаптера (Protocol) | Superset | DataLens (Public API RPC) |
|---|---|---|
| `ensure_database` | `POST /api/v1/database/` (`clickhousedb://…`) | `createConnection` → connection entry к CH/PG, **строковый connectionId** |
| `ensure_dataset` | `POST /api/v1/dataset/` (virtual SQL dataset) | `createDataset` из SQL-subselect над connection, **строковый datasetId** (TBD: Public API «dataset from SQL») |
| `create_chart` | `POST /api/v1/chart/` (`form_data`) | **`createEditorChart`** [exp] — Editor-чарт из конфига (Highcharts при HC=1), читает датасет; Wizard через API НЕ создаётся (§A.0). **Главный риск go/no-go 3.1** |
| `assemble_dashboard` | `POST /api/v1/dashboard/` (`position_json`) | `createDashboard` (**experimental**): tabs/widgets + selectors |
| native filters | `native_filter_configuration` | dashboard **selectors**, привязанные к полю датасета |

Ключевое следствие: **наш подход «один валидированный-SQL датасет на чарт» переносится 1:1** —
`generate_chart_sql` остаётся источником SQL, в DataLens он уходит в «dataset from SQL». Это значит,
что **joins не требуют отдельной обработки в DataLens-адаптере**: LEFT JOIN уже зашит в SQL (sqlgen.py:106),
DataLens видит плоский датасет с алиасами колонок (`column_alias`). Альтернатива (нативные dataset-relations
DataLens) сложнее и не даёт выгоды при SQL-датасетах — не берём.

### A.2 Маппинг 9 viz IR → DataLens Wizard

`Viz` IR (ir/spec.py:27) → тип DataLens + раскладка ролей по секциям Wizard.
Роли IR: `dimensions` / `series` / `rows` / `columns` / `measures` (ir/spec.py:96).

> После находки §A.0 эта таблица — **справка по viz-семантике** (что каким типом должно стать и какие
> роли куда ложатся). Реальный таргет адаптера — **Editor-конфиг** (Highcharts тип серии: line/column/bar/
> area/pie/heatmap/… + биндинг полей датасета), который воспроизводит те же соответствия. Колонка «DataLens viz»
> = и Wizard-имя, и Highcharts-аналог (совпадают по смыслу: line→`line`, bar→`column`, pie→`pie`, heatmap→`heatmap`).

| IR `viz` | DataLens viz | Раскладка ролей → секции | Деградация / примечание |
|---|---|---|---|
| `big_number` | **Indicator** | `measures[0]` → Measures | прямой |
| `line` | **Line chart** | `dimensions[0]`→X; `measures`→Y; `series`→Colors | прямой |
| `bar` | **Column chart** (вертикальный) | `dimensions[0]`→X; `measures`→Y; `series`→Colors | IR `bar`=вертикальный = DataLens **Column**, не «Bar» (тот горизонтальный) — зеркало Superset (`echarts_timeseries_bar`) |
| `stacked_bar` | **Column chart** + стек | `dimensions[0]`→X; `measures`→Y; `series`→Colors | стек включается наличием Colors/нескольких мер; 100%-стек = Normalized column |
| `area` | **Area chart** | `dimensions[0]`→X; `measures`→Y; `series`→Colors | прямой; stacked area нативно |
| `pie` | **Pie chart** | `dimensions[0]`→Colors; `measures[0]`→Measures | прямой (форм-валидация: 1 dim + 1 measure уже в Superset-пути) |
| `table` | **Table** | `group_columns()`+`measures`→Columns | прямой |
| `pivot` | **Pivot table** | `rows`→Rows; `columns`→Columns; `measures`→Measures | прямой (в SQL-чартах pivot недоступен — нужен dataset-based, у нас он и есть) |
| `heatmap` | Wizard: **нет** / Editor(HC): **есть** | `dimensions[0]`→x, `dimensions[1]`→y, `measures[0]`→value | **Editor-путь (§A.0) снимает деградацию**: Highcharts-серия `heatmap` (точки `[x,y,value]`) рисует декартов heatmap нативно. Деградация нужна ТОЛЬКО если HC недоступен (тогда Pivot+цвет/native, пометка в build log по инварианту §3.4). В Wizard декартова heatmap нет (только гео) — но Wizard не наш таргет |

Структурно это `VIZ_TYPE`-словарь как в form_data.py:15 + per-viz раскладка секций (аналог `build_form_data`).
Файл-кандидат: `auto_bi/adapters/datalens/wizard_config.py`.

### A.3 Фильтры дашборда → DataLens selectors

`spec.filters` (`DashboardFilter`) → dashboard-selector, привязанный к полю датасета. Логика
**scope-to-applicable** переносится дословно: `participating_chart_ids` (native_filters.py:54) уже
вычисляет по grain, какие чарты раскрывают колонку → селектор биндится только к ним; `filterType`

Семантика limit (top-N уезжает в чарт, а не в SQL) переносится тем же `apply_limit=False`-механизмом (sqlgen.py:78).

## B. SQL_GEN dialect seam — аудит под Greengage/Greenplum (задачи 3.3)

Greengage/Greenplum — PG-based; sqlglot имеет диалект `postgres`. Швы, где зашит ClickHouse:

| Локация | Что зашито | Минимальное изменение под PG |
|---|---|---|
| `agent/sqlgen.py:19` | `DIALECT = "clickhouse"` (модульная константа), `.sql(dialect=DIALECT)` (sqlgen.py:137) | Параметр `dialect: str` в `generate_chart_sql`, источник = `model.table(...).physical.engine` (CH→`"clickhouse"`, GP→`"postgres"`). AST-конструирование уже диалект-агностично (sqlglot транслирует) |
| `agent/sql_guard.py:12` | `DIALECT = "clickhouse"` для парсинга | тот же параметр диалекта в `guard_sql` |
| `agent/sql_guard.py:60` | trial-run обёртка CH-специфична: `SETTINGS max_execution_time = N` | per-engine обёртка: PG = `SET statement_timeout`/`SET LOCAL`; вынести в стратегию по движку |
| `agent/sql_guard.py:10`, `advisor/explain.py:11`, `advisor/core.py:14` | импорт `RunQuery` из `introspect.clickhouse` | перенести `RunQuery = Callable[[str], list[dict]]` (introspect/clickhouse.py:26) в нейтральный `introspect/base.py`, импортить оттуда (чистый рефактор, поведение не меняется, S4 НЕ триггерится) |

**Агрегации** (`_AGG_FUNC`, sqlgen.py:21) — `sum/avg/min/max/count` + `count(distinct)` — ANSI, работают в обоих движках без изменений.
**Фильтры/ordering/joins** — генерятся через sqlglot AST, диалект-нейтральны.

**Объём работы под второй диалект — невелик:** параметризовать диалект (2 константы), вынести trial-run-обёртку
в per-engine стратегию, перенести `RunQuery`. Это «дисциплина в швах» из ARCHITECTURE §1.1.

**S4-1. `TargetBI` enum: добавить `DATALENS = "datalens"`** (ir/spec.py:23). ✅ **СДЕЛАНО.**
- Зачем: spec должен таргетить второй BI; ARCHITECTURE §3.4 уже специфицирует `superset | datalens`.
- Риск: минимальный — добавление варианта enum, существующие spec'и (`target_bi` дефолт `superset`) не ломаются. Инертно до появления DataLens-адаптера.

**S4-2. BIAdapter ref `id: int` → `id: int | str`** (base.py — Database/Dataset/Chart/DashboardRef). ✅ **СДЕЛАНО.**
- Зачем: DataLens entryId — строки; `int` несовместим.
- **Уточнение рекомендации по итогам аудита кода:** взят вариант **`int | str`**, а НЕ `str`. Ref'ы
  потребляются только ВНУТРИ своего адаптера (в общий код не текут — `build`/`assemble` адаптер-специфичны),
  поэтому union не тянет проверок нигде, а Superset-путь остаётся **байт-в-байт прежним** (продолжает
  оперировать int — без конверсий, без правок в form_data/native_filters/position_json, live contract-тест
  не затрагивается). DataLens-адаптер положит строковый entryId. Это оптимальнее `str` (тот потребовал бы
  `int()`-конверсий на каждой Superset-wire-границе, которую нельзя live-верифицировать с Windows). ARCHITECTURE §3.5 обновлён.

**Не-S4, сделано сейчас:** перенос `RunQuery = Callable[[str], list[dict]]` в `introspect/base.py` ✅
(устранил дубль определения в clickhouse.py:26 И gaps.py:19; sql_guard/advisor.explain/advisor.core импортят
из нейтрального base — PG-путь больше не тянет CH-модуль ради generic-типа).

**Не-S4, оставлено на старт Phase 3 (нужен PG-стенд для live-verify):** параметр `dialect` в
`generate_chart_sql`/`guard_sql`; per-engine trial-run-обёртка; выбор advisor rule pack по движку;
