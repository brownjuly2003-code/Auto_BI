## Резюме

Phase 3 реализована добротно и соответствует плану: оба трека (движок Greenplum 3.3/3.4 + второй BI DataLens 3.1/3.2/3.5) закрыты, **P1-нарушений инвариантов нет**. Архитектурная дисциплина выдержана: DataLens-адаптер строит ВСЕ нативные форматы (connection/dataset/chart-`shared`/dashboard-`data`) детерминированным кодом из IR — LLM их не касается (инвариант 1); более того, схема датасета строится из IR без обращения к `validateDataset`/DB-интроспекции, что строго чище инварианта 1. Адаптер зеркалит `SupersetAdapter` шаг-в-шаг (5 методов Protocol + `build` оркестратор), ref-id `int|str` использован по назначению (S4-2). SQL для DataLens идёт через тот же `generate_chart_sql` с sqlglot-guard и per-engine диалектом (инвариант 3 не обойдён); секреты — только через `config.py`/`.env`, workbook-id дефолт намеренно несекретный и помечен. Идемпотентность (delete-then-create) реализована единообразно и аккуратно (connection — reuse-by-name, остальное — replace). Engine/advisor-шов корректен: per-engine dispatch rule-pack/EXPLAIN/диалекта по `physical.engine`, GP-интроспектор экранирует идентификаторы (`_ident`).

Главная слабость фазы — **не баг в адаптере, а его неподключённость к продукту**: весь build-путь (`cli.py`, `agent/pipeline.py`, API) жёстко завязан на `SupersetAdapter`, поле `spec.target_bi` (DATALENS существует в enum с Phase 3 prep) НИГДЕ не читается для выбора адаптера, и `validate_spec` его не проверяет. Адаптер существует и live-работает, но через продуктовый поток недостижим — exit-критерий «один и тот же spec собирается и в Superset, и в DataLens» доказан только контракт-тестом, дёргающим `DataLensAdapter` напрямую, не пользовательским путём. Это латентно и не ломает Phase 3 (детерминированная «DataLens-сборка» доказана), но Phase 4 («продукт», селектор BI в UI) на этом споткнётся первым.

Найдено: **0 P1 + 4 P2 + 7 P3**. Вердикт: **Phase 3 закрыть можно, Phase 4 открывать можно**; P2 закрыть до/в начале Phase 4 (особенно F1 — селектор BI это первая задача продуктового hardening, и F2 — селекторный partial-failure на живом стенде).

## Инварианты — что проверено как соблюдённое

- **#1 IR-first** — ✅. `chart_config.py`/`dataset.py`/`adapter.py` строят все native-блобы детерминированно; LLM эмитит только `DashboardSpec`. Схема датасета (`raw_schema`/`result_schema`) выводится из `ChartQuery` (роли/типы), не из БД — чище, чем требует инвариант.
- **#2 Spec валидируется до BI** — ✅ на Superset-пути (`compile_and_build` → `validate_spec` defense-in-depth перед адаптером). ⚠️ Для DataLens продуктового пути нет (см. F1), но это следствие неподключённости, а не обхода: контракт-тест строит из заведомо валидных spec'ов.
- **#3 SQL SELECT-only + guard + read-only** — ✅. DataLens subselect = тот же `generate_chart_sql` (sqlglot AST, `identify=True`); `compile_and_build` гоняет `LiveSQLValidator` (EXPLAIN+LIMIT). GP-интроспектор `_ident()` экранирует имена. Значения фильтров идут через `exp.Literal` (инъекции нет — см. ниже).
- **#4 Уточнения только из report** — вне диффа фазы (промпты не трогались).
- **#5 Advisor advisory-only** — ✅. `advisor/core.py` возвращает findings, GP-правила нигде не блокируют build.
- **#6 Fields-first** — вне диффа.
- **#7 Версия Superset запинена** — не задето (DataLens-стенд тоже зафиксирован образами compose, но это вне репо; см. F7-P3).
- **#8 Промпты мержатся с eval** — ✅ формально не задет: шаблоны промптов в диффе НЕ менялись (DataLens BI-агностичен через IR — golden не нужны, что верно). Eval расширен GP-кейсами (`GP_ADVISOR_CASES`/`GP_GOLDEN_CASES`), прогнан живьём.

Инъекция через label/имена/SQL: значения `QueryFilter.value` → `exp.Literal` (sqlglot экранирует); меры/измерения квалифицируются через `_dim_column`/`measure_alias`; `safe_entry_name` санитизирует только entry-имя (не значения) — реальной инъекции через chart/dataset/column-метки в DataLens-блобы не нашёл (имена попадают в JSON-поля, не в SQL).

## Findings

**F1 [P2] `auto_bi/cli.py:76,131,261` + `auto_bi/agent/pipeline.py:20-31,56-65` + `auto_bi/ir/validate.py` (нет проверки target_bi)** — DataLens-адаптер не подключён ни к одному продуктовому пути. `build_dashboard`/`compile_and_build` принимают `adapter: SupersetAdapter` (конкретный тип, не `BIAdapter`), `cli.py` во всех трёх местах инстанцирует `SupersetAdapter`, API-builder тоже. Поле `spec.target_bi: TargetBI` (значение `DATALENS` добавлено в Phase 3 prep) НИКОГДА не читается для выбора адаптера, и `validate_spec` его не валидирует. Следствия: (а) spec с `target_bi="datalens"` молча соберётся в Superset — «построенное ≠ заявленному», ровно тот класс молчаливого расхождения, что инвариант 2 запрещает на входе (ср. F2 audit Phase 1 про дроп фильтров); (б) exit-критерий «один spec → обе BI» доказан только контракт-тестом, дёргающим `DataLensAdapter` напрямую, не через пайплайн; (в) сигнатуры адаптеров разошлись — `SupersetAdapter.build(spec, model)` vs `DataLensAdapter.build(spec)` (model инжектится в конструктор), `assemble_dashboard` тоже расходится по аргументам, так что простой подменой типа их не свапнуть. Почему важно: первая задача Phase 4 — продуктовый селектор BI в UI (F8 audit Phase 2 — мёртвый `#bi-target`); без диспетчера адаптеров она нереализуема, а латентный «датасет в чужую BI» — реальный риск, как только селектор начнёт писать `target_bi`. Рекомендация: ввести фабрику `make_adapter(target_bi, settings, model) -> BIAdapter`, типизировать пайплайн на Protocol `BIAdapter` (а не `SupersetAdapter`), сверять `spec.target_bi` с выбранным таргетом в `validate_spec` или до build; выровнять `build`/`assemble_dashboard` сигнатуры под общий контракт (или явно вынести оркестратор из Protocol, как уже сделано для `build`).

**F2 [P2] `auto_bi/adapters/datalens/adapter.py:425-444` (build) + `:373-421` (assemble_dashboard)** — Частичный отказ build оставляет workbook в несогласованном состоянии без отката. `build` идёт connection→(per-chart: delete dataset→create→delete widget→create chart)→delete dash→create dash. Если упадёт, например, `create_chart` 3-го чарта (живой charts-engine транзиентно 500) — первые 2 датасета+чарта уже созданы и СТАРЫЕ одноимённые уже удалены (delete-then-create необратим в пределах прогона), дашборд не собран. В отличие от Superset (где `assemble_dashboard` линкует чарты в конце и старые сущности переиспользуются/обновляются, а не удаляются), здесь delete предшествует create на КАЖДОМ шаге, поэтому упавший ребилд уничтожает предыдущую рабочую версию dataset/widget, не создав замену. Транзиентные 5xx живого стенда — нормальный режим (реверс-спека прямо жалуется на флапающий туннель/память Mac). Связка с F1 audit Phase 2 (упавшая сборка невосстановима) усугубляет: следующий ретрай снова сделает getWorkbookEntries→(miss, т.к. удалили)→create — то есть в итоге сойдётся, но окно «дашборд сломан, старые чарты стёрты» открыто между прогонами. Рекомендация: документировать необратимость явно; рассмотреть отложенный delete (создать новое имя с суффиксом, удалить старое ТОЛЬКО после успешной сборки дашборда — как Superset собирает в новый дашборд), либо порядок «create-all → delete-stale → assemble»; как минимум — тест на partial-failure и предупреждение в build-логе, что ребилд не атомарен.

**F3 [P2] `auto_bi/adapters/datalens/adapter.py:283-301` (_find_entry_id) — false-positive в shared demo-workbook** — Идемпотентный lookup ищет entry по `getWorkbookEntries(scope, filters={name})` и матчит хвост `key` casefold. Workbook = `z4wtz6tg5194o` — это **OpenSource Demo workbook**, который (по реверс-спеке §5) содержит **84 готовых демо-чарта + 1 демо-dash + demo-датасеты**. `_find_entry_id` матчит по ТОЧНОМУ имени (casefold), так что прямой коллизии с демо-именами почти нет — НО: (а) если демо-entry случайно носит то же имя, что санитизированный `safe_entry_name(spec.title)` (например, обобщённое «Dashboard» / кириллический титул), `_delete_if_exists` УДАЛИТ чужой демо-entry перед создания своего — деструктивно и тихо; (б) casefold-матч на хвосте `key` означает, что два разных Auto_BI-титула, схлопывающихся в одно `safe_entry_name` (диакритика/пунктуация → пробел), будут считаться одним entry и затирать друг друга. Риск умеренный (имена Auto_BI начинаются с предсказуемых паттернов, демо — англоязычные дашборды), но цель — писать в ОБЩИЙ workbook с чужими данными, а delete необратим. Рекомендация: писать в выделенный workbook «Auto_BI» (ARCHITECTURE §3.5 прямо предписывает «сервисный аккаунт с правами только на выделенную папку/workspace Auto_BI») вместо demo; либо префиксовать ВСЕ Auto_BI entry-имена служебным маркером (`[Auto_BI] …`) и в lookup/delete ограничиваться только ими, чтобы исключить затирание чужих entry. Как минимум — задокументировать, что delete ограничен Auto_BI-именами и demo-workbook временный.

**F4 [P2] `auto_bi/adapters/datalens/dataset.py:175-185` (_field_aggregation) — re-aggregation идентичен только при grain dataset == grain chart** — Все меры в датасете получают `aggregation="sum"` (identity над пред-агрегированной строкой). Это верно ТОЛЬКО пока чарт группирует ровно по grain датасета (один датасет на чарт, grain = `group_columns()` чарта) — тогда SUM над одной строкой/группу = сама строка. Допущение нигде не защищено: `ensure_dataset` публичен и в контракт-тесте зовётся напрямую, `build_chart_shared` не сверяет, что placeholders чарта используют ровно `group_columns()` датасета. Если когда-нибудь чарт ляжет на датасет с более тонким grain (или AVG/MIN/MAX-мера попадёт в чарт, агрегирующий грубее) — `"sum"` молча даст НЕВЕРНОЕ число (сумма средних, сумма максимумов). Сейчас латентно (1 датасет=1 чарт=1 grain соблюдается в `build`), но «sum для всех мер» — мина под будущий шаринг датасетов/композитные чарты. Зеркальный Superset-путь имеет ту же uniform-SUM семантику (`form_data._adhoc_metric`), но там это тоже завязано на «датасет = grain чарта». Рекомендация: добавить assert/инвариант «dataset grain == chart group_columns» в `create_chart` (или хотя бы docstring-контракт + тест, что AVG-мера в чарте грубее grain даёт ошибку, а не тихий sum); либо маппить `aggregation` на исходный agg меры там, где это identity-safe.
